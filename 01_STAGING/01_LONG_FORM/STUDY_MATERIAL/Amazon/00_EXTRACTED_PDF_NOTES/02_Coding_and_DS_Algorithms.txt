# Coding, Data Structures, and Algorithms

Core coding expectation
Amazon expects production-like code in interviews:
- syntactically correct
- not pseudocode
- scalable
- robust
- readable
- maintainable
- well-tested
- close to production-ready under interview constraints

Phone interview coding
Typical environment:
- one coding problem
- language of your choice
- live coding
- no IDE/compiler
- interviewer expects you to explain as you code

Coding interview process

1. Restate the problem
Confirm inputs and expected output.

2. Ask clarifying questions
Examples:
- Can input be empty?
- Are values unique?
- Are values sorted?
- Can numbers be negative?
- What are the constraints?
- Do we optimize for time or memory?
- Is mutation allowed?
- Are duplicates possible?
- Should I return any valid solution or all solutions?
- What should happen on invalid input?

3. Identify examples
Walk through at least one normal case and one edge case.

4. Propose brute force
Briefly state the simplest approach and complexity.

5. Improve
Introduce better data structure or algorithm.

6. Explain tradeoff
State why this approach is appropriate.

7. Code
Write simple, modular, readable code.

8. Test
Run through:
- normal case
- edge case
- empty input
- single element
- duplicate values
- negative values
- large input
- invalid input if relevant

9. Complexity
State time and space complexity.

10. Improvements
Explain what you would improve with more time.

Production-like coding rules

A. Use descriptive variable names
Avoid:
- a, b, c, x unless obvious
Prefer:
- left, right
- start, end
- count_by_value
- visited
- queue
- candidate
- result

B. Modularize when useful
Use helper functions when they reduce complexity:
- dfs()
- bfs()
- is_valid()
- build_graph()
- binary_search()
- expand_from_center()
- merge()

C. Avoid unnecessary cleverness
Simple correct code beats obscure code.

D. Handle edge cases explicitly
Amazon prep repeatedly emphasizes edge cases.

E. Add tests
Interviewers expect you to scan code for errors and add test cases.

F. Explain complexity
Big-O is expected in nearly every technical interview.

Big-O knowledge

Be ready to state:
- time complexity
- space complexity
- worst case vs average case if relevant
- recursion stack space
- preprocessing vs query cost
- amortized cost
- impact of input dimensions, e.g. O(V + E), O(mn), O(n log k)

Common complexities:
- array scan: O(n)
- hash map lookup: average O(1)
- sorting: O(n log n)
- heap push/pop: O(log n)
- BFS/DFS graph: O(V + E)
- binary search: O(log n)
- dynamic programming grid: O(mn)
- pairwise comparison: O(n^2)
- backtracking: often exponential

Testing template
After writing code, say:
"I will test a normal case, a boundary case, and a failure/no-solution case."

Example categories:
- normal: [2,7,11,15], target 9
- edge: [], [1], duplicate values
- stress: large n, skewed tree, dense graph
- invalid: impossible target, missing key

Coding answer failure modes
- no clarification
- no brute-force baseline
- premature optimization
- no complexity
- no tests
- vague variable names
- hidden assumptions
- failure to handle empty inputs
- incorrect recursion base case
- off-by-one binary search error
- using global mutable state unnecessarily
- not explaining tradeoffs
